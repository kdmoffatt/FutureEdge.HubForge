# Portal Internationalization (i18n) Plan and Usage

This document defines how to add and maintain language support in HubForge portal apps.

## Goal

Enable user-selectable languages across all portal navigation, labels, and feature pages.

## Recommended Stack

- `react-i18next`
- `i18next`
- `i18next-browser-languagedetector`

## Directory Layout

Add locale files under:

- `apps/portal/app/locales/en.json`
- `apps/portal/app/locales/es.json`

Add i18n bootstrap under:

- `apps/portal/app/lib/i18n.ts`

## Translation Key Pattern

Use stable key namespaces:

- `nav.dashboard`
- `nav.users`
- `settings.theme.title`
- `jobs.table.status`

Avoid hardcoded strings in portal routes; always call translator function.

## Registering a New Language

1. Create new JSON file under `apps/portal/app/locales/<lang>.json`.
2. Add language metadata to i18n initialization resources.
3. Expose language in header selector.
4. Persist selected language in user preference or local storage.

## Tenant and User Defaults

Recommended precedence:

1. User preference (`settings` module key `preferences.language`)
2. Tenant default (`settings` module key `localization.defaultLanguage`)
3. App fallback (`en`)

## Developer Workflow

When creating pages or modules:

1. Add translation keys in `en.json` and `es.json`.
2. Replace literal text in route components with i18n lookups.
3. Add keys for table headers, actions, empty states, modals, and errors.

## Validation Checklist

- Language selector changes visible text without page reload
- Selection persists across navigation and refresh
- Missing keys fallback to English
- New module pages include translation keys before merge
